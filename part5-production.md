## 9. 프로덕션 아키텍처

이 장에서는 PAIF 기반 AI 서비스를 프로덕션 환경에서 운영하기 위한 아키텍처 패턴과 고려사항을 다룹니다. 고가용성, 재해복구, 멀티테넌시, 보안 등 엔터프라이즈 환경에서 필수적인 요소들을 설명합니다.

### 9.1 프로덕션 환경 개요

#### 9.1.1 개발 환경과 프로덕션 환경의 차이

AI 플레이그라운드 내에서 개발과 프로덕션은 **동일한 기술 스택**을 사용하지만, 운영 요구사항이 다릅니다.

#### 9.1.2 프로덕션 요구사항

| 요구사항 | 설명 | PAIF/PAIS 대응 |
|---------|------|---------------|
| **가용성** | 서비스 중단 최소화 | Model Endpoint 다중 Replica, VKS HA |
| **확장성** | 트래픽 증가 대응 | 자동 스케일링, GPU 동적 할당 |

| **복구** | 장애 시 빠른 복구 | Harbor 모델 백업, pgvector 백업 |
| **감사** | 규정 준수, 추적성 | API 로깅, VCF Operations 연동 |

### 9.2 고가용성(HA) 아키텍처

#### 9.2.1 HA가 필요한 구성요소

프로덕션 환경에서 고가용성을 확보해야 하는 구성요소:

| 구성요소 | HA 방식 | 장애 시 영향 |
|---------|--------|-------------|
| **Model Endpoint** | 다중 Replica | LLM 추론 불가 → 서비스 중단 |
| **Embedding Endpoint** | 다중 Replica | 검색 불가 → RAG 동작 불가 |
| **Agent API** | PAIS 내부 HA | RAG 서비스 중단 |
| **pgvector (DSM)** | DSM HA 구성 | 벡터 검색 불가 |
| **Harbor** | Harbor HA 구성 | 모델 Pull 불가 (신규 배포 영향) |
| **VKS Cluster** | Multi-node | 앱 Pod 장애 시 자동 복구 |

**VCF 기반 HA 계층 구조:**

PAIF 환경의 고가용성은 VCF 인프라부터 PAIS 서비스까지 3개 계층으로 보장됩니다.

| 계층 | 기술 | 보호 대상 | 장애 시나리오 | 복구 방식 |
|------|------|----------|--------------|----------|
| **Layer 1** | vSphere HA | ESXi 호스트, VM | 호스트 장애, 하드웨어 오류 | 다른 호스트에서 VM 자동 재시작 |
| **Layer 2** | VKS ReplicaSet | Kubernetes Pod | Pod 크래시, 노드 장애 | 다른 노드에서 Pod 자동 재생성 |
| **Layer 3** | PAIS 자동복구 | Model Endpoint | vLLM 프로세스 오류, GPU 장애 | Health Check 실패 시 Pod 교체 |

> **💡 핵심:** 각 계층이 독립적으로 동작하여 다중 방어선을 형성합니다.
> ESXi 호스트가 장애나도 vSphere HA가 VM을 복구하고, 
> Pod가 비정상이면 VKS가 재생성하며, 
> 모델 추론이 실패하면 PAIS가 해당 Replica를 제외합니다.

#### 9.2.2 Model Endpoint HA 구성

Model Endpoint는 프로덕션에서 **가장 중요한 HA 대상**입니다. GPU 리소스가 필요하므로 비용과 가용성의 균형이 필요합니다.

```

│   │                    ML API Gateway (PAIS)                            │  │
│   │                    • 로드 밸런싱                                     │  │
│   │                    • Health Check                                   │  │
│   │                    • 자동 Failover                                  │  │
│   └───────────────────────────────┬─────────────────────────────────────┘  │
│                                   │                                        │
│              ┌────────────────────┼────────────────────┐                  │
│              │                    │                    │                  │
│              ▼                    ▼                    ▼                  │
│   ┌─────────────────┐  ┌─────────────────┐  ┌─────────────────┐           │
│   │   Replica #1    │  │   Replica #2    │  │   Replica #3    │           │
│   │                 │  │                 │  │                 │           │
│   │   vLLM Pod      │  │   vLLM Pod      │  │   vLLM Pod      │           │
│   │   + GPU (A100)  │  │   + GPU (A100)  │  │   + GPU (A100)  │           │
│   │                 │  │                 │  │                 │           │
│   │   Status: ●     │  │   Status: ●     │  │   Status: ●     │           │
│   │   Active        │  │   Active        │  │   Active        │           │
│   │                 │  │                 │  │                 │           │
│   │   ESXi Host A   │  │   ESXi Host B   │  │   ESXi Host C   │           │
│   └─────────────────┘  └─────────────────┘  └─────────────────┘           │
│                                                                           │
│   장애 시나리오:                                                           │
│   • Replica #1 장애 → Gateway가 자동으로 #2, #3로 라우팅                    │

**ML API Gateway란?**

PAIS가 제공하는 AI 서비스 전용 API 게이트웨이입니다. 내부적으로 **Envoy Proxy** 기반으로 구현되어 있습니다.

| 기능 | 설명 | VCF/PAIF 연동 |
|------|------|--------------|
| 인증/인가 | OIDC 토큰 검증, RBAC 적용 | VCF Automation OIDC Provider 연동 |
| 로드 밸런싱 | 다중 Replica 간 요청 분배 | Round-robin, Least-connections |
| Health Check | 정상 Replica만 라우팅 | vLLM `/health` 엔드포인트 연동 |
| API 호환성 | OpenAI API 형식 지원 | `/v1/chat/completions`, `/v1/embeddings` |
| Rate Limiting | 사용량 제한으로 남용 방지 | 네임스페이스/사용자별 설정 가능 |

**일반 API Gateway와의 차이점:**

| 항목 | 일반 API Gateway | PAIS ML API Gateway |
|------|-----------------|-------------------|
| 기본 타임아웃 | 30초 | 60초+ (LLM 추론 시간 고려) |
| 스트리밍 | 제한적 지원 | SSE(Server-Sent Events) 네이티브 지원 |
| 스케일링 연동 | 수동 설정 | GPU 사용률 기반 자동 스케일링 |
| 모델 라우팅 | 없음 | 모델명 기반 Endpoint 라우팅 |

**Replica 수 권장 가이드:**

| 환경 | 최소 Replicas | 권장 Replicas | 비고 |
|------|--------------|---------------|------|
| 개발/테스트 | 1 | 1 | 비용 최소화 |
| 스테이징 | 1 | 2 | 프로덕션 유사 환경 검증 |
| 프로덕션 (일반) | 2 | 3 | N+1 이상 권장 |
| 프로덕션 (미션 크리티컬) | 3 | 3+ | 동시 장애 대비 |

> **💡 비용 고려:** 각 Replica는 GPU를 점유하므로, Replicas 증가 = GPU 비용 증가입니다. 자동 스케일링을 활용하여 최소 Replicas를 유지하고, 부하 시에만 증가시키는 전략을 권장합니다.

#### 9.2.3 pgvector (DSM) HA 구성

Knowledge Base의 벡터 데이터를 저장하는 pgvector도 HA 구성이 필요합니다.

```

│   │                    DSM (Data Services Manager)                      │  │
│   │                                                                     │  │
│   │   DSM은 PostgreSQL 인스턴스의 HA를 자동으로 관리합니다.                │  │
│   │                                                                     │  │
│   │   ┌─────────────────┐      ┌─────────────────┐                      │  │
│   │   │   Primary       │      │   Standby       │                      │  │
│   │   │   PostgreSQL    │ ───▶ │   PostgreSQL    │                     │  │
│   │   │   + pgvector    │ 동기 │   + pgvector    │                      │  │
│   │   │                 │ 복제 │                 │                      │  │
│   │   │   Read/Write    │      │   Read Only     │                     │  │
│   │   └─────────────────┘      └─────────────────┘                     │  │
│   │            │                        │                              │  │
│   │            │         자동 Failover  │                              │  │
│   │            │         (Primary 장애 시)                             │  │
│   │            ▼                        ▼                              │  │
│   │   ┌─────────────────────────────────────────────────────────────┐  │  │
│   │   │              Virtual IP (VIP)                               │  │  │
│   │   │              앱에서는 VIP로 접속 → 장애 시 자동 전환           │  │  │
│   │   └─────────────────────────────────────────────────────────────┘   │  │
│   │                                                                     │  │
│   └─────────────────────────────────────────────────────────────────────┘  │
│                                                                             │
│   DSM HA 설정 시 고려사항:                                                   │
│   • 동기 복제 (Sync) vs 비동기 복제 (Async): 데이터 일관성 vs 성능             │

│   • 백업: 정기적인 스냅샷/덤프 백업 필수                                       │

#### 9.2.4 VKS 클러스터 HA

앱을 호스팅하는 VKS 클러스터의 HA 구성:

| 구성요소 | HA 방식 | 설명 |
|---------|--------|------|
| **Control Plane** | 3개 노드 (자동) | VKS가 자동으로 3개 Control Plane 노드 배포 |
| **Worker Node** | Multi-node | 최소 3개 Worker 노드 권장 |
| **Pod** | Deployment + ReplicaSet | Replicas 2 이상 설정 |
| **Service** | LoadBalancer | NSX가 자동으로 로드 밸런싱 |

### 9.3 재해복구(DR) 전략

#### 9.3.0 DR 핵심 개념 정의

DR(재해복구) 전략을 이해하기 위해 먼저 핵심 개념을 정의합니다.

**RTO (Recovery Time Objective) - 복구 시간 목표**

장애 발생 시점부터 서비스가 정상 복구될 때까지 허용되는 **최대 시간**입니다.

- 예시: "RTO 1시간" = 장애 발생 후 1시간 내에 서비스 재개 필요
- 비즈니스 영향: RTO가 짧을수록 더 많은 인프라 투자 필요
- AI 서비스 관점: Model Endpoint 재시작, GPU 할당, 모델 로딩 시간 모두 포함

**RPO (Recovery Point Objective) - 복구 시점 목표**

장애 발생 시 **데이터 손실을 허용하는 최대 시점**입니다. 마지막 백업 이후의 데이터는 손실될 수 있습니다.

- 예시: "RPO 15분" = 최대 15분치 데이터 손실 허용
- 비즈니스 영향: RPO가 짧을수록 더 빈번한 백업/복제 필요
- AI 서비스 관점: Knowledge Base 인덱스, 대화 이력, 사용자 피드백 데이터 등

**DR 패턴 개요**

| 패턴 | 정의 | VCF 구현 기술 | RTO | RPO | 비용 |
|------|------|--------------|-----|-----|------|
| **Pilot Light** | 핵심 구성요소만 DR 사이트에 최소 가동 | Harbor Replication, pgvector 백업, PAIS 설정 GitOps | 수 시간 | 수 시간 | 낮음 |
| **Warm Standby** | 축소 규모로 DR 사이트 상시 가동 | VCF Stretched Cluster 또는 독립 사이트 + 실시간 복제 | 수십 분 | 수 분 | 중간 |
| **Active-Active** | 양 사이트 동시 운영, 자동 Failover | NSX ALB GSLB + Harbor 양방향 복제 + pgvector 동기 복제 | 수 분 이내 | 0 (동기) | 높음 |

- **Pilot Light**: 가스레인지의 "파일럿 불꽃"처럼 최소한만 켜두고, 필요 시 전체를 점화
- **Warm Standby**: DR 사이트가 "예열된 상태"로 대기, 스케일업만 하면 즉시 서비스 가능
- **Active-Active**: 양쪽 모두 "활성 상태", 한쪽 장애 시 다른 쪽이 전체 부하 흡수

#### 9.3.1 DR 시나리오

PAIF 환경에서 고려해야 할 재해 시나리오:

| 시나리오 | 범위 | 복구 방법 |
|---------|------|----------|
| **Pod 장애** | 단일 Pod | Kubernetes 자동 재시작 |
| **Node 장애** | 단일 ESXi 호스트 | vSphere HA로 VM 재시작 |
| **Cluster 장애** | 전체 VKS 클러스터 | 백업에서 복구 또는 재배포 |
| **사이트 장애** | 전체 데이터센터 | DR 사이트로 Failover |

**패턴 3: Active-Active (최고 가용성)**

**GSLB (Global Server Load Balancing)란?**

GSLB는 지리적으로 분산된 여러 데이터센터 간에 트래픽을 분배하는 글로벌 로드밸런서입니다.

**VCF 환경에서의 GSLB 구현:**

| 옵션 | 구현 방식 | 특징 |
|------|----------|------|
| **NSX ALB (Avi) GSLB** | NSX Advanced Load Balancer의 GSLB 기능 | VCF 네이티브 통합, DNS 기반 |
| **외부 GSLB** | F5 BIG-IP DNS, Cloudflare, AWS Route 53 등 | 멀티클라우드 지원, 기존 인프라 활용 |

**AI 서비스에서 GSLB 사용 시 고려사항:**

- **세션 일관성**: 동일 사용자의 대화가 같은 사이트로 라우팅되어야 함 (Sticky Session 또는 Session ID 기반 라우팅)
- **모델 버전 동기화**: 양 사이트의 Model Endpoint가 동일한 모델 버전을 서빙해야 함
- **Knowledge Base 동기화**: pgvector 데이터가 양 사이트에서 일관되어야 함

#### 9.5.3 스케일링 고려사항

| 고려사항 | 설명 | 권장 |
|---------|------|------|
| **Cold Start** | 새 Replica 시작 시 모델 로딩 시간 | Min Replicas ≥ 1 유지 |
| **GPU 가용성** | GPU 리소스가 부족하면 스케일 아웃 불가 | GPU Pool 여유 확보 |
| **스케일 속도** | 너무 빠르면 불안정, 너무 느리면 응답 지연 | 점진적 스케일링 설정 |
| **비용** | Replicas 증가 = GPU 비용 증가 | Max Replicas 제한 설정 |

**Cold Start 상세**

Cold Start는 새로운 Model Endpoint Replica가 시작될 때 발생하는 초기 지연입니다. AI 서비스에서 특히 중요한 이유는 모델 로딩에 상당한 시간이 소요되기 때문입니다.

**Cold Start 발생 단계:**

| 단계 | VCF/PAIF 구성요소 | 소요 시간 | 설명 |
|------|------------------|----------|------|
| 1. Pod 스케줄링 | VKS Scheduler | 수 초 | Kubernetes가 적절한 노드에 Pod 배치 |
| 2. GPU 할당 | vGPU Manager / DirectPath I/O | 수 초 ~ 수십 초 | GPU 리소스 할당 및 드라이버 초기화 |
| 3. 컨테이너 시작 | Container Runtime | 수 초 | vLLM 컨테이너 이미지 시작 |
| 4. 모델 로딩 | vLLM + GPU Memory | 수십 초 ~ 수 분 | Harbor에서 모델 Pull → GPU 메모리 적재 |
| 5. 워밍업 | vLLM | 수 초 | 첫 추론 요청 처리 (CUDA 커널 컴파일 등) |

**모델 크기별 Cold Start 예상 시간 (A100 80GB 기준):**

| 모델 크기 | 예상 시간 | 주요 병목 |
|----------|----------|----------|
| 8B (Llama 3.1 8B) | 30초 ~ 1분 | 모델 로딩 |
| 70B (Llama 3.1 70B) | 2분 ~ 5분 | 모델 로딩 + 멀티 GPU 초기화 |

**vGPU vs DirectPath I/O가 Cold Start에 미치는 영향:**

| GPU 할당 방식 | Cold Start 영향 | 이유 |
|--------------|----------------|------|
| vGPU | 약간 느림 | vGPU Manager의 가상화 오버헤드 |
| DirectPath I/O | 약간 빠름 | GPU 직접 액세스, 단 vMotion 불가 |

**Cold Start 완화 전략:**

1. **Min Replicas ≥ 1 유지**: 항상 최소 1개 Replica가 워밍 상태 유지
2. **워밍업 요청**: 배포 직후 더미 요청을 보내 사전 로딩 완료
3. **적정 모델 선택**: 8B로 충분하면 70B 사용 금지 (로딩 시간 5배 차이)
4. **이미지 캐싱**: 노드에 모델 이미지 사전 Pull (ImagePullPolicy: IfNotPresent)
